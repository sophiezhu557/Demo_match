# ABC 公益导师匹配系统 Demo

这是一个用于方案演示的导师匹配系统 Demo。前端为原生 HTML/CSS/JavaScript，后端为 Python 标准库 HTTP 服务，数据从 `input/mentors.csv` 和 `input/students.csv` 初始化到 SQLite。

## 本地运行

```bash
python server.py
```

然后打开：

```text
http://127.0.0.1:4173/
```

## 部署到 Render

1. 将本目录上传到 GitHub 仓库。
2. 在 Render 创建 Web Service，并连接该 GitHub 仓库。
3. 如果使用 `render.yaml`，Render 会自动读取以下配置：
   - Build Command: `pip install -r requirements.txt`
   - Start Command: `python server.py`
   - `HOST=0.0.0.0`
4. 在 Render 的 Environment Variables 中设置：
   - `DEMO_ACCESS_CODE`: 演示访问码(abc2026)，用于保护 demo 页面和 API。

## 可选：持久化 SQLite 数据

默认情况下，云端数据库会由 CSV 初始化，适合短期演示。如果希望 Render 重启或重新部署后保留操作记录，可以在 Render 后台添加 Persistent Disk，并设置：

```text
DEMO_DB_PATH=/var/data/abc_mentor_demo.sqlite3
```

同时将 Persistent Disk 挂载到：

```text
/var/data
```

## 重要说明

这个项目是 Demo，不是正式生产系统。当前“登录”用于演示三种身份视角，并不是真实账号体系。正式上线前需要补充真实认证、权限控制、日志审计和数据备份。


# 匹配模式说明
这个网站的功能是：学员可以在上面选自己感兴趣的老师然后申请成为导师的学生。
导师是各行各业的，学员是ABC公益组织的。在开始匹配导师之前，学员和导师都会提交问卷，透露一些自己的信息。  

导师信息：  
结构化信息：姓名、毕业院校、行业、职位、感兴趣的方向。  
非结构化信息：曾经参与过的项目经历、关注的话题、其他的补充留言。  
学员信息：  
结构化信息：姓名、院校、专业、感兴趣的方向。  
非结构化信息：在系统匹配前是否已经有意向导师、在系统匹配前是否已和某位导师提前约定过、过往经历、补充留言。  

然后，系统首先把导师的信息整合到网站上，提供一些可选的筛选分类，供学员查看选择。

#### 匹配机制：

当前 demo 采用“三轮匹配”机制，整体原则是：第一轮由学员提交志愿、导师进行预匹配，系统在第一轮截止时统一结算；第二轮为补录；第三轮由管理员人工匹配。

目前默认规则如下：

1. 导师默认最多接收 3 位学员，但导师可以在导师端自定义自己的名额上限。
2. 第一轮中，每位学员可以一次性提交第一、第二、第三志愿导师。学员可以不填满三个志愿，但提交后不可修改。
3. 第一轮中，导师看到所有把自己列为第一（绿色）、第二（蓝色）或第三（黄色）志愿的学员申请，并可以选择将学员加入“预匹配”。
4. 第一轮期间，导师的选择不是最终匹配结果，而是“预匹配中”。系统会在第一轮截止时统一结算。
5. 如果同一名学员被多位导师预匹配，系统优先将其匹配给志愿顺位更高的导师。例如，第一志愿导师和第二志愿导师都选择了同一名学员，则最终匹配给第一志愿导师。
6. 对于未最终匹配成功的导师，导师端仍会保留历史申请记录；相关申请会变灰，并显示原因，例如“该学生已与更高志愿的导师匹配”。
7. 一轮匹配结束后，导师尚未接受或拒绝的申请会被视为“超时申请”。
8. 如果学员在某一轮未匹配成功，系统会向学员展示失败原因，并让学员选择是否继续参加下一轮匹配或退出匹配。

#### 匹配步骤：

**第一轮：志愿提交与统一结算**

1. 学员提交第一、第二、第三志愿导师，可以不填满三个志愿。
2. 导师查看申请自己的学员，并选择是否将学员加入“预匹配”。
3. 第一轮期间，预匹配不代表最终匹配成功。
4. 管理员点击“结束第一轮”后，系统统一结算：
   - 若学员被多位导师预匹配，优先匹配给更高志愿导师；
   - 若导师未在截止前处理申请，则该申请记为超时；
   - 若导师明确拒绝，则失败原因记为需求不匹配。
5. 第一轮未匹配成功的学员可以选择：
   - 参加第二轮补录；
   - 或退出匹配。

**第二轮：补录申请**

1. 第二轮只面向第一轮未成功匹配、且选择继续匹配的学员。
2. 学员只能定向申请一位尚未招满的导师。
3. 导师可以接受或拒绝该申请。
4. 第二轮结束后：
   - 未被导师处理的申请记为超时；
   - 被导师拒绝的申请记为需求不匹配。
5. 第二轮仍未匹配成功的学员可以选择：
   - 进入第三轮管理员人工匹配；
   - 或退出匹配。

**第三轮：管理员人工匹配**

1. 第三轮面向第二轮后仍未匹配成功、且选择进入人工匹配的学员。
2. 管理员根据学员和导师问卷、导师剩余名额、申请状态等信息进行人工匹配。
3. 管理员完成处理后，可以结束第三轮，并向相关学员发送最终通知。

#### 参与方：

本项目有 3 个参与方：学员、导师、管理员。

**学员端：**

学员可以浏览所有导师信息，查看自己的问卷信息，提交第一轮志愿，查看申请状态和系统通知。在匹配失败后，学员可以看到失败原因，例如匹配超时或需求不匹配，并选择继续下一轮匹配或退出匹配。学员也可以通过反馈窗口与管理员沟通。

**导师端：**

导师可以查看自己的问卷信息和所有申请自己的学员材料。第一轮中，导师可以看到将自己列为第一、第二、第三志愿的学员，并选择将学员加入预匹配。导师也可以自定义自己的名额上限。系统会显示导师当前名额、已匹配人数、是否满额。对于未最终匹配成功的历史申请，导师端仍会保留记录，并显示原因。导师也可以通过反馈窗口与管理员沟通。

**管理员端：**

管理员可以查看导师和学员提交的问卷信息、匹配状态、申请状态和异常反馈。管理员可以结束每一轮匹配，查看学员失败原因，处理第三轮人工匹配，并在特殊情况下解除已经完成的导师-学员匹配。管理员端也提供测试辅助功能，例如撤回第一轮结算，方便 demo 阶段反复测试第一轮结束前后的状态变化。
