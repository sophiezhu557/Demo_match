# ABC 公益导师匹配系统 Demo

这是一个用于方案演示的导师匹配系统 Demo。前端为原生 HTML/CSS/JavaScript，后端为 Python 标准库 HTTP 服务，数据从 `input/mentors.csv` 和 `input/students.csv` 初始化到 SQLite。

## 本地运行

```bash
python server.py
```

然后打开：

```text
http://127.0.0.1:4173/
```

## 部署到 Render

1. 将本目录上传到 GitHub 仓库。
2. 在 Render 创建 Web Service，并连接该 GitHub 仓库。
3. 如果使用 `render.yaml`，Render 会自动读取以下配置：
   - Build Command: `pip install -r requirements.txt`
   - Start Command: `python server.py`
   - `HOST=0.0.0.0`
4. 在 Render 的 Environment Variables 中设置：
   - `DEMO_ACCESS_CODE`: 演示访问码(abc2026)，用于保护 demo 页面和 API。

## 可选：持久化 SQLite 数据

默认情况下，云端数据库会由 CSV 初始化，适合短期演示。如果希望 Render 重启或重新部署后保留操作记录，可以在 Render 后台添加 Persistent Disk，并设置：

```text
DEMO_DB_PATH=/var/data/abc_mentor_demo.sqlite3
```

同时将 Persistent Disk 挂载到：

```text
/var/data
```

## 重要说明

这个项目是 Demo，不是正式生产系统。当前“登录”用于演示三种身份视角，并不是真实账号体系。正式上线前需要补充真实认证、权限控制、日志审计和数据备份。


# 匹配模式说明
这个网站的功能是：学员可以在上面选自己感兴趣的老师然后申请成为导师的学生。
导师是各行各业的，学员是ABC公益组织的。在开始匹配导师之前，学员和导师都会提交问卷，透露一些自己的信息。

导师信息：\n
结构化信息：姓名、毕业院校、行业、职位、感兴趣的方向。\n
非结构化信息：曾经参与过的项目经历、关注的话题、其他的补充留言。\n
学员信息：\n
结构化信息：姓名、院校、专业、感兴趣的方向。\n
非结构化信息：在系统匹配前是否已经有意向导师、在系统匹配前是否已和某位导师提前约定过、过往经历、补充留言。\n

然后，系统首先把导师的信息整合到网站上，提供一些可选的筛选分类，供学员查看选择。


#### 匹配机制：
目前默认的两个规则（后续可以视情况改动）：
1. 一个导师最多收3个学员，每个学员只申请1个导师。
2. 默认在收集完了所有学员的问卷以后进行匹配，所以不按先来后到进行。
学员先浏览所有导师的信息，然后选择一位导师进行申请。
每个导师都有一个选择池，导师从选择池里选择要不要接受来申请的学生。一个老师的选择池里人数不超过8人，老师最多就从这8人里选。
可能出现多于8个学员同时申请1个导师的情况，这种情况下，先用算法或者AI把匹配度最高的学员放到老师的选择池子里，其余人视为不匹配，被系统自动筛掉。但如果专家愿意接收更多学员，也可以选择看到所有申请ta的学员。

#### 匹配步骤：
1. 最优先：如果学员在问卷中对这个问题“在系统匹配前是否已和某位导师提前约定过“有指定的回答，优先把学员放进导师的选择池子里。如果此时选择池子就已经超过8人，则忽略第1、2步直接按照步骤3来。
2. 次优先：学员在问卷提到，在系统匹配前已有意向导师，且该导师的选择池子人数还未超过8人。
3. 完全按照双方问卷信息的匹配度来进行匹配。

#### 参与方：
本项目有3个参与方，一方是学员，一方是导师，一方是管理员。
学员需要：能看到所有导师信息、能查看申请进度、能看到自己提交过的问卷信息、有和管理员反馈情况的窗口；
导师需要：能看到最多8个申请者信息的选择池、如果愿意，能看到所有申请者的信息、能选择是否接收学生、能看到自己提交过的问卷信息、有和管理员反馈情况的窗口；
管理员需要：能看到双方提交的所有信息（问卷）、能看到学员和导师反馈的异常情况、能修改学员和导师的匹配情况（在某些特殊情况下）。在管理员的dashboard里，目前已经可以实现两个功能，第一是可以解除已经匹配的导师-学员，第二是可以撤销还尚未出结果，但已经提交的学员申请。
